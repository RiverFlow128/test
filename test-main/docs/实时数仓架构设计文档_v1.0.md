# 实时数仓架构设计文档
## 1. 架构概览

### 1.1 设计目标
构建一个高性能、高可靠、可扩展的实时数据仓库，为业务提供实时数据支持，满足运营分析、业务决策和实时监控的需求。

### 1.2 分层架构设计
```
┌─────────────────────────────────────────────────────────────────┐
│                         ADS 应用数据层                           │
│        (实时报表、运营看板、游戏数据大屏、实时预警)                 │
│                (阿里云Kafka)                                     │
└─────────────────────────────────────────────────────────────────┘
                                ▲
┌─────────────────────────────────────────────────────────────────┐
│                         DWS 汇总数据层                           │
│            (轻度聚合、多维汇总、分钟/小时级指标)                   │
│                (阿里云Kafka)                                     │
└─────────────────────────────────────────────────────────────────┘
                                ▲
┌─────────────────────────────────────────────────────────────────┐
│                         DWD 明细数据层                           │
│              (数据清洗、标准化、维度关联、去重)                    │
│                (阿里云Kafka)                                     │
└─────────────────────────────────────────────────────────────────┘
                                ▲
┌─────────────────────────────────────────────────────────────────┐
│                         DIM 维度数据层                           │
│              (维度表管理、缓慢变化维处理、维度一致性保障)            │
│                (阿里云Kafka)                                     │
└─────────────────────────────────────────────────────────────────┘
                                ▲
┌─────────────────────────────────────────────────────────────────┐
│                         ODS 原始数据层                           │
│              (MySQL CDC实时同步、原始数据落地)                    │
│                (阿里云Kafka)                                     │
└─────────────────────────────────────────────────────────────────┘
                                ▲
┌─────────────────────────────────────────────────────────────────┐
│                         数据源 (MySQL)                           │
│                    (用户库、订单库、游戏事件库)                    │
└─────────────────────────────────────────────────────────────────┘
```

### 1.3 数据流转说明
数据流转路径清晰明了：从MySQL的各个业务库（用户库、订单库、游戏事件库等）通过CDC技术实时同步到ODS层，ODS层的数据经过处理后进入DIM层进行维度管理，然后DWD层从ODS层获取原始数据并与DIM层的维度数据进行关联，经过清洗和加工后，再到DWS层进行聚合计算，最后在ADS层生成各种报表和看板，直接服务于运营和业务决策。

### 1.4 核心设计考量
- **数据倾斜处理**：针对开服瞬间大量用户登录导致的数据倾斜问题，在Flink中采用keyBy结合随机前缀的方式分散热点
- **乱序数据处理**：考虑到移动端网络不稳定导致的日志上报延迟，设置30秒的Watermark，确保数据准确性
- **状态管理**：针对Flink State可能过大的问题，使用RocksDB作为状态后端，并配置合理的内存和磁盘参数
- **数据回溯**：利用Kafka的offset机制，支持通过重置消费组offset来重新消费数据，应对代码错误等场景

## 2. 各层详细设计

### 2.1 ODS层 (Operational Data Store)
ODS层作为原始数据层，主要负责将MySQL中的数据实时同步过来，尽量保持数据的原貌，为后续处理提供基础。

| 项目 | 说明 |
|------|------|
| **数据来源** | MySQL Binlog（通过Flink CDC实时捕获） |
| **处理逻辑** | 原样落地，不做业务转换，保留原始数据的所有特征 |
| **存储策略** | 存储到阿里云Kafka的Topic中，确保数据安全且可追溯 |
| **数据特点** | 包含完整的增删改操作，反映数据的完整变更历史 |
| **技术实现** | 使用Flink SQL + MySQL CDC连接器，配置阿里云Kafka连接参数 |

### 2.2 DIM层 (Dimension Data Store)
DIM层作为维度数据层，主要负责维度表的管理、缓慢变化维的处理和维度一致性的保障。

| 项目 | 说明 |
|------|------|
| **数据来源** | ODS层的阿里云Kafka Topic |
| **处理逻辑** | 维度表构建、缓慢变化维处理、维度一致性保障等操作 |
| **存储策略** | 处理后存储到阿里云Kafka中，确保维度数据的实时性和一致性 |
| **技术实现** | 使用Flink SQL从Kafka读取数据，进行维度处理后写回Kafka |

**主要处理内容**：
- **维度表构建**：从ODS层抽取维度数据，构建完整的维度表
- **缓慢变化维处理**：根据业务需求采用适当的SCD策略（Type 1/Type 2/Type 3）
- **维度一致性保障**：确保维度数据在整个数据仓库中的一致性
- **维度缓存**：实现维度数据的缓存机制，提高维度关联性能







### 2.3 DWD层 (Data Warehouse Detail)
DWD层作为明细数据层，主要进行数据清洗和标准化，为后续的聚合分析做准备。

| 项目 | 说明 |
|------|------|
| **数据来源** | ODS层的阿里云Kafka Topic |
| **处理逻辑** | 数据清洗、格式统一、维度关联、去重等操作 |
| **存储策略** | 处理后存储到阿里云Kafka中，部分中间结果暂存在Flink的State中 |
| **技术实现** | 使用Flink SQL从Kafka读取数据，进行清洗和转换后写回Kafka |

**备注**：这里的维度关联需要注意，尽量使用异步IO或预加载维度表，避免因为维度表过大导致处理延迟增加。

**主要处理内容**：
- **字段标准化**：统一时间格式（如转为标准时间戳）、枚举值映射（如将状态码转为可读状态）等
- **数据去重**：基于主键或业务键去除重复数据，确保数据唯一性
- **维度扩展**：关联DIM层的用户、商品、渠道等维度信息，丰富数据语义
- **异常数据过滤**：剔除无效或异常数据，如空值、超出合理范围的值等

### 2.4 DWS层 (Data Warehouse Summary)
DWS层作为汇总数据层，主要进行轻度聚合，生成各种维度的汇总指标，为上层应用提供分析基础。

| 项目 | 说明 |
|------|------|
| **数据来源** | DWD层处理后的明细数据（阿里云Kafka） |
| **处理逻辑** | 按时间窗口（分钟、小时等）和业务维度（渠道、服务器等）进行聚合计算 |
| **存储策略** | 存储到阿里云Kafka中，利用其高吞吐量和可靠性 |
| **技术实现** | 使用Flink SQL进行聚合计算，结果存储到Kafka |

**聚合维度**：
- **时间维度**：分钟、小时、天等不同粒度
- **业务维度**：游戏ID、渠道、服务器、地区等
- **用户维度**：用户类型、等级、活跃度等

### 2.5 ADS层 (Application Data Store)
ADS层作为应用数据层，直接面向业务应用，提供最终的数据产品和服务。

| 项目 | 说明 |
|------|------|
| **数据来源** | DWS层的聚合结果和DWD层的明细数据（阿里云Kafka） |
| **处理逻辑** | 计算业务指标、生成报表、数据可视化 |
| **存储策略** | 存储到阿里云Kafka中，确保高可用性和实时性 |
| **数据特点** | 直接服务应用，数据结构友好，查询性能高 |
| **技术实现** | 使用Flink SQL从Kafka读取数据，生成最终报表后写回Kafka |

## 3. 技术实现方案

### 3.1 核心实现细节
- **CDC同步**：使用Flink MySQL CDC连接器实时捕获MySQL Binlog，确保数据实时同步
- **数据处理**：采用Flink SQL进行数据处理，简化开发复杂度，提高开发效率
- **状态管理**：使用RocksDB作为状态后端，配置合理的内存和磁盘参数
- **窗口计算**：根据业务需求，设置合理的窗口大小和滑动步长
- **数据写入**：采用批量写入策略，提高写入性能
- **阿里云Kafka集成**：配置SASL_SSL安全认证，确保数据传输安全
- **精确一次语义**：使用Flink的exactly-once语义，确保数据处理的一致性
- **高并发处理**：配置合理的并行度，使用数据分片和负载均衡提高并发能力
- **异步IO**：实现异步维度关联，提高系统响应速度和吞吐量

### 3.3 高并发与异步IO优化

#### 高并发处理策略
1. **并行度配置**：
   - 设置全局并行度为16，根据服务器资源和数据量调整
   - 为不同的连接器设置合适的并行度，确保资源充分利用

2. **数据倾斜处理**：
   - 对热点数据添加随机前缀，分散处理压力
   - 使用自定义分区策略，确保数据均匀分布
   - 优化窗口计算，避免数据集中处理

3. **Kafka写入优化**：
   - 配置批量写入参数：`sink.buffer-flush.max-rows` = 1000
   - 设置写入间隔：`sink.buffer-flush.interval` = 5000ms
   - 优化Kafka客户端参数：`batch.size` = 16384, `linger.ms` = 5

4. **状态管理优化**：
   - 使用RocksDB作为状态后端，支持大状态
   - 配置检查点间隔为60秒，确保容错性
   - 优化RocksDB内存管理和缓存设置

#### 异步IO实现
1. **维度关联的异步化**：
   - 使用Flink的`AsyncDataStream`进行维度表查询
   - 实现异步维度查询函数，避免同步等待
   - 添加维度数据缓存，减少重复查询

2. **外部系统交互的异步化**：
   - 使用异步Kafka写入，提高写入吞吐量
   - 批量提交减少网络IO次数
   - 利用Flink的背压机制，在系统负载高时自动调节处理速度

#### 游戏行业特殊优化
1. **高峰时段处理**：
   - 动态资源调整，根据游戏开服、活动等高峰时段调整并行度
   - 预计算热门游戏和活动数据，减少实时计算压力
   - 限流机制，避免系统过载

2. **数据模型优化**：
   - 按游戏ID、服务器等维度进行分区，提高查询效率
   - 对热门游戏和高频操作单独处理，避免影响整体系统
   - 优化时序数据处理，适应游戏日志的时序特性

### 3.2 阿里云Kafka配置
**Topic设计**：
- ODS层：ods-device-users-relations, ods-game-record, ods-user-online-status

## 4. 运维与监控

### 4.1 监控体系
- **技术指标监控**：
  - Flink作业监控：延迟、吞吐量、CPU/内存使用率、状态大小
  - 阿里云Kafka监控：消息积压、消费延迟、分区状态、SASL认证状态
  - MySQL监控：连接数、查询性能、主从同步状态

- **业务指标监控**：
  - 核心业务指标：DAU、新用户注册数、活跃用户数、用户留存率
  - 指标波动监控：设置合理的波动阈值，及时发现异常
  - 趋势分析：监控指标的长期趋势，预测未来走势

- **数据质量监控**：
  - 数据完整性：监控数据缺失情况
  - 数据一致性：监控不同层级数据的一致性
  - 数据及时性：监控数据从产生到分析的延迟

- **告警机制**：
  - 阈值告警：设置合理的告警阈值
  - 趋势告警：监控指标的异常趋势
  - 关联告警：多个指标关联分析，减少误报

### 4.2 部署与运维
- **部署方式**：
  - 使用Docker容器化部署，确保环境一致性
  - 使用Kubernetes进行容器编排，实现自动扩缩容
  - 配置资源限制，确保系统稳定运行

- **CI/CD**：
  - 建立自动化部署流程，使用Jenkins或GitLab CI
  - 实现代码提交自动构建和测试
  - 配置灰度发布机制，降低部署风险

- **版本管理**：
  - 使用Git进行代码版本管理
  - 建立分支管理策略，确保代码质量
  - 记录版本变更日志，便于问题追溯

- **故障处理**：
  - 建立故障应急预案，明确处理流程
  - 实现数据备份和恢复机制
  - 定期进行故障演练，提高应对能力

### 4.3 监控工具推荐
- **Prometheus**：用于收集和存储监控指标
- **Grafana**：用于可视化监控数据
- **Alertmanager**：用于管理和发送告警
- **ELK Stack**：用于日志收集和分析
- **Zabbix**：用于系统级监控
