# 实时数仓架构设计文档 v2.0

> **版本说明**：v2.0 版本在 v1.0 基础上进行了全面的规范化和完善，主要更新包括：时区统一、数据链路补全、清洗逻辑规范化等。

## 1. 架构概览

### 1.1 设计目标
构建一个高性能、高可靠、可扩展的实时数据仓库，为业务提供实时数据支持，满足运营分析、业务决策和实时监控的需求。

### 1.2 分层架构设计
```
┌─────────────────────────────────────────────────────────────────┐
│                         ADS 应用数据层                           │
│        (实时报表、运营看板、游戏数据大屏、实时预警)                 │
│                       (PolarDB/MySQL)                           │
└─────────────────────────────────────────────────────────────────┘
                                ▲
┌─────────────────────────────────────────────────────────────────┐
│                         DWS 汇总数据层                           │
│            (轻度聚合、多维汇总、分钟/小时级指标)                   │
│                    (阿里云Kafka → PolarDB)                       │
└─────────────────────────────────────────────────────────────────┘
                                ▲
┌─────────────────────────────────────────────────────────────────┐
│                         DWD 明细数据层                           │
│              (数据清洗、标准化、空值过滤、类型转换)                │
│                       (阿里云Kafka)                              │
└─────────────────────────────────────────────────────────────────┘
                                ▲
┌─────────────────────────────────────────────────────────────────┐
│                         DIM 维度数据层                           │
│              (维度表管理、缓慢变化维处理、维度一致性保障)           │
│                   (MySQL CDC → PolarDB)                         │
└─────────────────────────────────────────────────────────────────┘
                                ▲
┌─────────────────────────────────────────────────────────────────┐
│                         ODS 原始数据层                           │
│              (MySQL CDC实时同步、原始数据落地)                    │
│                       (阿里云Kafka)                              │
└─────────────────────────────────────────────────────────────────┘
                                ▲
┌─────────────────────────────────────────────────────────────────┐
│                         数据源 (MySQL)                           │
│      (用户库user_basic、订单库ordercenter、资产库assetscenter)    │
└─────────────────────────────────────────────────────────────────┘
```

```sql
SET 'execution.checkpointing.externalized-checkpoint-retention' = 'RETAIN_ON_CANCELLATION';

```

#### 微批优化配置
```sql

```

## 2. 数据模块详细设计

### 2.1 ODS层 (Operational Data Store)

#### 2.1.1 设计原则
- **原样落地**：不做业务转换，保留原始数据特征
- **变更捕获**：通过 `row_status` 元数据字段记录操作类型
- **过滤删除**：WHERE条件过滤删除操作 `row_status <> '-D'`

|------|----------|-----------|-------------|

| 提现订单 | ordercenter-test | withdraw_order | flink_withdraw_order_data_sync |
| 第三方代付 | ordercenter-test | third_withdraw_payment_order | flink_third_withdraw_payment_order_data_sync |
| 游戏记录 | gamecenter-test | game_record_* | flink_game_order_data_sync |
| 奖励记录 | - | reward_record | flink_reward_record_data_sync |
| 金额操作 | admincenter-test | amount_operation_record | flink_amount_operation_record_data_sync |
| 代理统计 | assetscenter-test | user_proxy_statistic | flink_user_proxy_statistic_data_sync |

#### 2.1.4 ODS层表数量说明

> **为什么ODS层只有5个文件，而DWD层有7个文件？**

ODS层只包含通过 **Flink CDC 从MySQ L接入** 的数据源。有些数据源不经过ODS层，而是由业务系统直接写入Kafka：

| 数据源 | 接入方式 | ODS文件 | 说明 |
|--------|---------|----------|------|
| recharge_order | MySQL CDC | ✅ 有 | 充值订单从MySQ L同步 |
| withdraw_order | MySQL CDC | ✅ 有 | 提现订单从MySQ L同步 |
| game_record | MySQL CDC | ✅ 有 | 游戏记录从MySQ L同步 |
| amount_operation_record | MySQL CDC | ✅ 有 | 金额操作从MySQ L同步 |
| user_proxy_statistic | MySQL CDC | ✅ 有 | 代理统计从MySQ L同步 |
| **reward_record** | **业务直写Kafka** | ❌ 无 | 奖励记录由业务系统实时推送 |
| **user_online_status** | **业务直写Kafka** | ❌ 无 | 用户在线状态由业务系统实时推送 |

**数据接入方式对比：**
```
方式1: MySQL CDC接入（5个）
  MySQL表 → Flink CDC → ODS Kafka → DWD Kafka → DWS

方式2: 业务直写Kafka（2个）
  业务系统 → ODS Kafka → DWD Kafka → DWS
  （无需Flink CDC，数据不在MySQL中）
```

### 2.3 DWD层 (Data Warehouse Detail)

#### 2.3.1 设计原则（v2.0 规范）
- **空值过滤**：必须过滤所有用于计算 `statistic_time` 的时间字段
- **类型转换**：统一数据类型（如 INT → BOOLEAN）
- **状态过滤**：只保留有效状态的记录

#### 2.3.2 清洗逻辑规范
```sql
-- 标准WHERE子句模板
WHERE id IS NOT NULL
  AND update_time IS NOT NULL   -- 必须：防止statistic_time为NULL
  AND [业务状态过滤条件];
```

#### 2.3.3 DWD层模块清单
| 模块 | 源Topic | 目标Topic | 清洗要点 |
|------|---------|-----------|----------|
| 充值订单 | flink_recharge_order_data_sync | dwd_recharge_order | id/update_time非空 |
| 提现订单 | flink_withdraw_order_data_sync | dwd_withdraw_order | 状态=COMPLETED/REJECTED, update_time非空 |
| 游戏记录 | flink_game_order_data_sync | dwd_game_record | 状态=settle, order_no/update_time非空 |
| 奖励记录 | flink_reward_record_data_sync | dwd_reward_record | id/user_id/channel_id/update_time/create_time非空, draw_flag转BOOLEAN |
| 金额操作 | flink_amount_operation_record_data_sync | dwd_amount_operation_record | 状态=COMPLETE, id/update_time非空 |
| 代理统计 | flink_user_proxy_statistic_data_sync | dwd_user_proxy_statistic | id/user_id/channel_id/statistic_time非空 |
| 用户在线状态 | flink_user_online | user_online_status_record (PolarDB) | user_id/online_time/create_time非空，**直接写入PolarDB** |

#### 2.3.4 DWD层SQL文件清单
```
src/main/resources/sql/dwd/
├── recharge_order_dwd.sql
├── withdraw_order_dwd.sql
├── game_record_dwd.sql
```

#### 2.3.5 DWD层与DWS层数量差异说明

> **为什么DWD层有7个文件，而DWS层只有6个？**

`kafka_to_user_online_status_record.sql`（用户在线状态）**不需要DWS层**。

| 模块 | 是否需要DWS | 原因 |
|------|----------|------|
| 充值/提现/游戏/奖励等 | 需要 | 业务数据需要按用户/渠道汇总统计 |
| 用户在线状态 | 不需要 | 它本身就是明细记录，不需要汇总 |

```
其他业务流程：
  DWD (每笔订单明细) → DWS (按用户汇总) → 输出统计表
  例：「用户A 充值100」+「用户A 充值200」→「用户A 总充值300」

用户在线状态流程：
  DWD (每次上线/下线记录) → 直接写入PolarDB
  例：「用户A 10:00上线」「用户A 12:00下线」→ 保持明细，不汇总
```

### 2.4 DWS层 (Data Warehouse Summary)

#### 2.4.1 设计原则
- **多维聚合**：按用户、渠道、时间等维度聚合
- **空值处理**：SELECT中使用COALESCE与GROUP BY保持一致
- **类型一致**：聚合条件使用正确的类型（如 `draw_flag = TRUE`）

### 2.5 ADS层 (Application Data Store)

#### 2.5.1 设计原则
- **面向应用**：直接服务于前端报表、运营看板
- **维度精简**：仅保留应用层需要的维度，去除细粒度字段
- **模块拆分**：按业务模块独立统计，便于维护和扩展

#### 2.5.2 与mlbb的架构差异

> **mlbb是一张大宽表，当前项目拆成了3张表，但都支持日级+累计统计**

| 对比项 | mlbb | 当前项目 |
|--------|------|----------|
| 文件数 | 1个 | 3个 |
| 日级统计表 | today_user_aggregation_reward_bet_bounty_statistic | today_user_game_statistic + today_user_reward_statistic + today_user_bounty_statistic |
| 累计统计表 | user_aggregation_reward_bet_bounty_statistic | user_game_statistic + user_reward_statistic + user_bounty_statistic |
| 设计思路 | 所有指标合并到一张大宽表 | 按业务模块拆分成独立表 |

**mlbb大宽表包含的字段：**
```
today_user_aggregation_reward_bet_bounty_statistic:
  - total_reward_*                  ← 奖励统计（70+字段）
  - third_party_total_*             ← 三方游戏统计
  - bounty_total_*                  ← 赏金统计
```

**当前项目拆分后：**
```
today_user_game_statistic      ← 游戏统计（独立表）
today_user_reward_statistic    ← 奖励统计（独立表）
today_user_bounty_statistic    ← 赏金统计（独立表）
```

#### 2.5.3 拆分的原因

1. **维护性更好** - 各模块独立迭代，不会相互影响
2. **灵活性更高** - 可以单独启用/停用某个模块的统计
3. **排查更方便** - 出问题时可以快速定位到具体模块
4. **扩展性更强** - 新增业务模块时只需新增文件，不用改现有表结构

#### 2.5.4 ADS层文件清单

| 文件 | 数据源 | 日级表 | 累计表 |
|------|--------|--------|--------|
| today_user_game_statistic.sql | DWS层 | today_user_game_statistic | user_game_statistic |
| today_user_reward_statistic.sql | DWD层 | today_user_reward_statistic | user_reward_statistic |
| today_user_bounty_statistic.sql | DWD层 | today_user_bounty_statistic | user_bounty_statistic |

```
src/main/resources/sql/ads/
├── today_user_game_statistic.sql       # 用户游戏统计（日级+累计）
├── today_user_reward_statistic.sql     # 用户奖励统计（日级+累计）
└── today_user_bounty_statistic.sql     # 用户赏金统计（日级+累计，占位）
```

#### 2.5.5 ADS层数据源说明

> **为什么有的ADS从DWS读，有的直接从DWD读？**

| ADS表 | 数据源 | 原因 |
|--------|--------|------|
| today_user_game_statistic | DWS层 | DWS已经按「用户」统计过了，ADS直接复用 |
| today_user_reward_statistic | DWD层 | DWS是按「活动类型」统计的，但ADS要按「奖励类型」统计，维度不同 |
| today_user_bounty_statistic | DWD层 | 赏金业务没有DWS层，只能从DWD读 |

## 3. 完整数据血缘关系

### 3.1 充值订单模块
```
MySQL: recharge_order
    ↓ (MySQL CDC)
ODS: flink_recharge_order_data_sync (Kafka)
    ↓ (清洗: id/update_time非空)
DWD: dwd_recharge_order (Kafka)
    ↓ (聚合)
DWS: today_user_recharge_statistic_dws (MySQL)
     today_channel_recharge_statistic (MySQL)
```

### 3.2 提现订单模块（双源表）
```
MySQL: withdraw_order ──────────────────► flink_withdraw_order_data_sync (Kafka)
MySQL: third_withdraw_payment_order ───► flink_third_withdraw_payment_order_data_sync (Kafka)
    ↓                                        ↓
    ↓ (清洗: 状态过滤, update_time非空)        ↓ (直接用于DWS聚合)
DWD: dwd_withdraw_order (Kafka)              ↓
    ↓                                        ↓
    └────────────┬───────────────────────────┘
                 ↓ (聚合 + LEFT JOIN)
DWS: today_user_withdraw_statistic_dws (MySQL)
     today_channel_withdraw_statistic (MySQL)
     today_user_withdraw_statistic_finance_increment (MySQL) ← 需要JOIN两张表
```

### 3.3 游戏记录模块
```
MySQL: game_record_* (1024分表)
    ↓ (MySQL CDC, 正则匹配)
ODS: flink_game_order_data_sync (Kafka)
    ↓ (清洗: status=settle, order_no/update_time非空)
DWD: dwd_game_record (Kafka)
    ↓ (多维聚合)
DWS: today_user_game_statistics_dws (MySQL)
     today_user_hour_game_ug_statistics_data (MySQL)
     today_channel_game_category_statistic (MySQL)
     today_channel_game_provider_statistic (MySQL)
     today_channel_game_record_statistic (MySQL)
     game_record (MySQL) ← 原始记录落地
```

### 3.4 奖励记录模块
```
Kafka: flink_reward_record_data_sync (JSON格式，非debezium)
    ↓ (清洗: 空值过滤, draw_flag INT→BOOLEAN)
DWD: dwd_reward_record (Kafka, debezium-json)
    ↓ (多维聚合, draw_flag使用TRUE/FALSE比较)
DWS: today_user_reward_statistics_dws (MySQL)
     today_channel_reward_statistic (MySQL)
     today_channel_activity_statistic (MySQL)
     reward_record (MySQL) ← 原始记录落地
```

### 3.5 金额操作模块
```
MySQL: amount_operation_record
    ↓ (MySQL CDC)
ODS: flink_amount_operation_record_data_sync (Kafka)
    ↓ (清洗: status=COMPLETE, id/update_time非空)
DWD: dwd_amount_operation_record (Kafka)
    ↓ (聚合)
DWS: today_user_custom_amount_statistics_data (MySQL)
     today_channel_custom_amount_statistic (MySQL)
```

### 3.6 代理统计模块
```
MySQL: user_proxy_statistic
    ↓ (MySQL CDC)
ODS: flink_user_proxy_statistic_data_sync (Kafka)
    ↓ (清洗: id/user_id/channel_id/statistic_time非空)
DWD: dwd_user_proxy_statistic (Kafka)
    ↓ (聚合)
DWS: today_channel_proxy_statistic (PolarDB)
```

### 3.7 用户在线状态模块
```
业务系统直写 Kafka: flink_user_online
    ↓ (清洗: user_id/online_time/create_time非空)
DWD: user_online_status_record (PolarDB)
    （无DWS层，直接写入终端表）
```

### 5.4 架构升级：user_proxy_statistic 链路

#### 5.4.1 与mlbb的差异

| 对比项 | mlbb | 当前项目 |
|--------|---------|----------|
| 文件 | user_proxy_statistic_to_db_mlbb | ods/user_proxy_statistic_to_kafka.sql + dwd + dws |
| 架构 | 表对表直接同步 | 完整数仓分层 |
| 中间层 | 无Kafka | 有ODS/DWD Kafka |
| 输出 | polar_user_proxy_statistic (维度表) | today_channel_proxy_statistic (统计表) |

#### 5.4.2 数据链路对比

```
mlbb（DIM层同步）:
  MySQL.user_proxy_statistic → CDC → PolarDB.user_proxy_statistic
  （简单的表对表同步，无聚合计算）

当前项目（完整数仓分层）:
  MySQL.user_proxy_statistic
      ↓ CDC
  ODS: flink_user_proxy_statistic_data_sync (Kafka)
      ↓ 清洗
  DWD: dwd_user_proxy_statistic (Kafka)
      ↓ 聚合
  DWS: today_channel_proxy_statistic (PolarDB)
  （支持渠道级别的代理统计聚合）
```

> 注：mlbb的 `user_proxy_statistic_sync.sql` 仍然保留在 dim/ 目录下，用于维度表同步。两套链路并行存在，互不影响。
